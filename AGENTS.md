# Clipin

现代化 macOS 剪贴板管理器。Rust 后端 + SwiftUI 前端。

## 架构

- **Rust Core**（`rust/`）：存储（SQLite+FTS5）、搜索、数据模型
- **Swift UI**（`Clipin/`）：SwiftUI 界面、系统集成（menu bar、快捷键、剪贴板监控）
- **UniFFI Bridge**：Rust ↔ Swift 互操作，自动生成绑定代码

## 构建

```bash
# 首次构建 Rust + 生成 Swift bindings
./scripts/build-rust.sh

# 生成 Xcode 项目
xcodegen generate

# 编译
xcodebuild -project Clipin.xcodeproj -scheme Clipin -configuration Release -destination 'platform=macOS,name=Any Mac' build

# Rust 测试
cd rust && cargo test --lib
```

## 关键路径

- `rust/src/lib.rs` — Rust 入口 + UniFFI 导出（ClipinCore API）
- `rust/src/storage.rs` — SQLite + FTS5 存储层（v4：含 ocr_text 列）
- `Clipin/App/AppDelegate.swift` — Menu bar + NSPanel + 生命周期
- `Clipin/Views/MainPanel.swift` — Raycast 风格双栏布局
- `Clipin/Services/ClipboardMonitor.swift` — 剪贴板轮询
- `Clipin/Services/HotKeyService.swift` — ⌘+Shift+V 全局快捷键
- `Clipin/Services/PasteService.swift` — 模拟 Cmd+V 粘贴
- `Clipin/Generated/` — UniFFI 自动生成（.gitignore）
- `scripts/build-rust.sh` — 构建脚本
- `project.yml` — xcodegen 配置

## 决策

- **图片 OCR**：使用 Apple Vision Framework `VNRecognizeTextRequest`（macOS 原生，零依赖，支持中英文）。图片写盘后在同一 `Task.detached` 内串行 OCR，结果写入 `clip_items.ocr_text` 列。FTS5 索引同步覆盖 ocr_text，搜索图片文字与搜索文本完全统一。列表 preview 通过 SQL 层 `COALESCE(NULLIF(ocr_text,''),content)` 智能回退，无需改 Swift 模型。
- **Bridging Header**（非 modulemap）：解决 Xcode Explicit Module Build 下 UniFFI C 头文件导入问题
- **xcodegen**：代码化管理 Xcode 项目，.xcodeproj 不提交
- **Rust 预构建脚本必须声明输入/输出，不能长期关闭 dependency analysis**：如果 `Build Rust Core` 每次 build 都全量跑，开发时噪音大、增量构建也会被拖慢。当前做法是在 `project.yml` 里显式声明 Rust 核心脚本、Cargo 清单、UniFFI 入口和现有 Rust 源文件为输入，静态库与生成的 Swift/FFI 文件为输出，并把 `basedOnDependencyAnalysis` 打开；这样 Xcode 只会在这些输入变化时重跑脚本。已知前提是新增 Rust 源文件时要同步更新这份输入列表，避免把新文件漏出增量图谱。
- **LSUIElement=true**：纯 menu bar app，不出现在 Dock
- **AppState: `@unchecked Sendable`**（非 `@MainActor`）：ClipinCore 内部用 Mutex 保证线程安全
- **AppDelegate: `@MainActor`**：所有 UI 交互在主线程，避免 Swift 6 actor isolation 错误
- **不沙盒**：CGEvent tap 需要辅助功能权限，沙盒限制太多
- **隐私感知采集**：`ConcealedType`（密码管理器）始终跳过，`TransientType` / `AutoGeneratedType`（临时/自动生成）由设置开关控制（默认跳过）。前者是不可关闭的安全红线，后者是降噪偏好。
- **设置窗口用普通 window level，不用 `.floating`**：`.floating` 会让设置窗口悬浮在所有 app 上方，导致 Cmd+W 发给其他 app 的 key window 而不是设置窗口。正确做法是 `NSApp.activate` 让 Clipin 成为 frontmost，设置窗口作为普通窗口随 app 焦点切换自然退到背后，和所有 macOS 偏好窗口行为一致。
- **主面板、动作面板、设置页和权限窗口都必须共享同一套 surface/control/selection 语法，不要各自长出独立皮肤**：真正要统一的不只是颜色 token，还包括 `section gutter / row chrome / content stage` 这套布局骨架。当前做法是把统一 chrome 下沉成共享 `ClipinSurfaceBackground` / `ClipinSelectableRowBackground` / `ClipinKeycap` 原件；主列表向动作面板看齐，先给 section 留 gutter，再在里面放 selected row；preview 右侧保留一层安静的内容 stage 和一个轻量 metadata block，但这两者必须是专用 surface role，而不是继续复用设置页那种 generic grouped 面，避免 detail 区重新长回“卡片套卡片”。
- **全 App 还要共享同一套 shell / hero / section intro 语法，不只是共享卡片底板**：如果 launcher、preferences、permission、onboarding 只共用表面材质，却各自手搓页头、图标英雄区和外层背景，用户感知上仍然会像四个产品。当前做法是把外层壳统一下沉成 `ClipinShellBackground`，把场景主图标和页头说明统一成 `ClipinSymbolOrb` / `ClipinSectionIntro`，让不同窗口在第一眼就读成同一家族。
- **小字号信息必须走专用 ink token，不能再直接套 macOS `tertiary/quaternary` 语义色**：`time label / section header / metadata label / shortcut key / empty-state hint` 这类 9-11pt 文案如果直接用系统 tertiary，会在浅色主题里掉到几乎不可读。当前做法是在 `ClipinPanelHierarchy` 里补一层 `support` 语义（`subdued / smallLabel / hint / placeholder`）并给选中态补 `dimInk`，让小字既能退场，也始终保持可读。
- **右侧预览区不要再做“头部 badge + 底部 details”两套语法，而应收成“上面纯内容、下面单行 footer rail”**：Clipin 的右侧 preview 首先是内容舞台，不是 inspector 表单；左侧列表已经给了类型线索，右侧再放类型 badge 只会抢来源和时间的位置。当前做法是把来源、相对时间、Pinned、类型相关摘要指标、复制次数统一收进内容下方的一条横向 footer rail，默认单行展示，放不下时横向滑动而不是自动隐藏；文本统计在 UI 层使用更接近用户感知的字符数，并用 `NaturalLanguage` 分词，URL 不再硬显示 `words`。
- **搜索区和底栏都要先读成“控制器”，不要长回一排散按钮，也不要把搜索条做成过高的 hero 横幅**：如果 filter pills、clear、actions、continuous paste、settings 都各自裸露在同一平面，界面会显得忙，也会削弱主动作；而如果搜索区垂直高度过大，又会抢走第一屏的内容舞台。当前做法是把搜索 filters 收进统一的 filter rail，把底栏辅助命令收进 command cluster，并把搜索输入、筛选胶囊和 header 上下节奏一起收成更接近 macOS compact toolbar 的紧凑控制条。
- **列表的类型图标要先成为“对象”再谈留白，不能裸悬在文本前面**：没有背景容器的图标会让整行缺少视觉锚点，即使数学上加了 inset，看起来也还是挤。当前做法是让非缩略图类型统一落到一层轻量 icon pill 里，并在 hover 时只做极小尺度反馈，这样重点更集中，列表节奏也更稳。
- **“不要贴边”不是简单加 padding，而是让对象本身形成稳定列宽**：如果 selected row、preview stage、metadata block 继续铺满各自容器，即使数学上有 inset，视觉上仍然会读成贴边。当前做法是在 section gutter 之上，再给列表 row 和右侧 detail 对象单独留 `object inset`，让选中项、预览舞台和信息块都像“摆在里面”的对象，而不是“铺在里面”的底色。
- **底栏是固定高度的 command strip，不能再让 Paste CTA 反向决定整体高度**：之前 footer 的高度被左侧主胶囊按钮撑大，导致有选中项和无结果时气质不统一。当前做法是先定义 strip 自己的最小高度和外边距，再把 CTA 收进这个高度里，同时压缩 icon 和垂直 padding，让 footer 始终像一条命令条而不是一颗大按钮拖着一整条底栏走。
- **“看起来贴边”往往是 shell 间距和对象宽度同时失衡，不是单独某个 inset 太小**：如果 header / content / footer 三段之间没有统一 spacing 体系，section surface 之间的影子和亮边也会互相挤压，最后即使对象自身有圆角也会显得满、乱，甚至在深色主题里出现被裁切的尖角感。当前做法是把主面板收回到统一的 `shellSectionInset / shellSectionGap / header-content-footer vertical rhythm`，同时降低 sidebar/detail 的阴影半径和位移，减少 section shadow 在底部和边缘的尖角残影。
- **设置窗口所有入口都必须落到同一个自定义 NSWindow，而不是一部分走 SwiftUI Settings scene、一部分走 AppDelegate 窗口**：否则会出现首击行为、鼠标交互、键盘上下文和窗口样式不一致，看起来像“设置页又坏了”。当前做法是把 `Command+,` 的 `.appSettings` 命令显式重定向到 `AppDelegate.openSettingsWindow()`，保留自定义 `ClipinHostingView` 的首次点击行为，彻底消除系统 Settings scene 和自定义窗口并存带来的交互分叉。
- **连续粘贴模式**：面板保持可见，失去 key window 时自动夺回焦点（通过 `ClipinPanel.resignKey` 回调 + 150ms 延迟让用户点击先完成）。用户可以在不同应用的输入框间点击定位光标，面板自动回焦后直接上下选择、回车粘贴，实现零额外操作的连续粘贴。通过 `didActivateApplicationNotification` 实时更新底栏目标应用名，粘贴时 `resolveTargetApp()` 实时查询 `frontmostApplication` 兜底。Esc 退出连续粘贴模式。常驻可见（不夺回焦点、可编辑浮动笔记）作为后续 feature。
- **设置页 sidebar 必须是原生 selection container，不要再用一排 `Button` 自己模拟键盘语义**：之前设置页能稳定响应 `↑↓`，本质上依赖的是 `List(selection:)` 这类原生单选容器；后面把 sidebar 重写成自定义 `Button` 列以后，键盘导航语义被抽空，才开始出现一层层窗口级兜底。当前做法是保留 `SettingsNavigationModel` 作为状态源，但把 sidebar 重新交回 `List(selection:)` 托管键盘选中，再在 row 内套自定义 chrome，这样既保留统一视觉，也回到系统稳定的导航行为。
- **设置页 sidebar 的“可键盘导航”和“看起来像 Clipin”必须拆层处理，不要让原生视觉绑架交互语义**：根因不是 `List` 本身，而是把原生 selection 语义和 AppKit 默认 sidebar 高亮混成了一件事。当前做法是继续用 `List(selection:)` 提供键盘/焦点/单选行为，行间距、row background、交替底色等尽量使用 SwiftUI 原生 modifier 表达；只有“关闭 macOS 默认选中高亮”这一处 SwiftUI 尚未暴露的能力，才通过极薄的 AppKit bridge 处理。这样设置页能和主面板共享同一套 chrome，同时不再回到手搓键盘导航。
- **设置页详情区要先建立“当前页在管什么”，再展开具体控件；相关设置应合并成完整表单组，而不是一排同权重碎卡片**：如果只保留标题、把开关和 picker 打散成多个独立 surface，视觉上虽然统一，信息层级却会变弱，用户需要自己猜每张卡之间的关系。当前做法是给每个 sidebar tab 增加页头说明，相关控件收回到少量 form group，再把不可关闭的安全说明、备份状态这类内容降到单独的 supporting/status surface；同时把 sidebar 列宽和 row object 宽度一起调大，让选中对象形成稳定列宽，不再显得短和挤。
- **设置页左侧导航要统一的是“选中对象的 object inset”，不是盲目拉宽整列 sidebar**：如果只把 sidebar 总宽度做大，而选中态高亮底板仍然离两侧边缘太远，视觉上还是会读成一颗偏短的胶囊，和右侧接近整卡宽度的内容对象不是同一种语言。当前做法是让 sidebar row 使用和主列表一致的 `ClipinChrome.listRowOuterInset` 作为外层 gutter，让 `selected row background` 更接近“摆在 surface 里的对象”而不是“浮在中间的小 pill”。
- **设置页的上下键问题要从“初始焦点”解，不要重新长回窗口级方向键路由**：现在设置 sidebar 的键盘语义已经交回原生 `List(selection:)`，真正还会让用户觉得“方向键失效”的根因，是打开窗口后 first responder 没落到原生 `NSTableView`。当前做法是在设置窗口真正成为 key window 后，把初始 responder 显式交给 sidebar table；这样既保留系统 `List` 的导航语义，也不会把设置页再拉回一套特例键盘监视器。
- **全 App 要统一的是 scene recipe，不是要求所有页面共用一块“detail 底板”**：Clipin 现在至少有 launcher / preferences / command palette / modal 四种场景；它们应共享同一套 token、shadow scale 和 surface role，但每个场景的容器配方可以不同。当前做法是把主面板右侧收回成直接浮在 shell 上的 `contentStage + metadata` 卡片列，不再额外套一层大 `detail` 背景；设置页和权限页则保留 `column` 容器，动作面板继续是 `floating` recipe。
- **“灵动”必须来自 scene state 编排，而不是在各个控件上零散加动画**：如果搜索、选中、动作面板、连续粘贴各自只在本地 view 里做 `scale/opacity`，界面只会变得更忙，不会更活。当前做法是把 launcher 的核心状态收口成 `ClipinSceneState`，再由 `shell / search / selected row / preview / footer strip / action palette` 共享一套 motion 语法（`ambient / focusShift / commandReveal / statePulse`）；这样状态变化会沿整条主路径一起流动，同时保留 `Reduce Motion` 的静态退化路径。
- **阴影不能再大过 panel gap，否则“没贴边也会看起来贴边”**：如果 section 之间的 gap 是 12pt，而 sidebar / column 的 shadow radius 还在 16-18pt，视觉上就会把间隙吃满，用户会读成“贴边、溢出甚至超框”。当前做法是把 section 级 shadow 收到 8/4，contentStage 收到 8/3，metadata/strip/grouped 收到 6/2-3，保证层级存在但不会把 panel gap 吃掉。
- **类型筛选切换也必须是全局导航，不应依赖搜索框焦点**：只在 `SearchBar` 内部拦截 `Tab` 会导致用户点击列表/预览区后，`Tab` 退化成默认焦点遍历，和“键盘优先”的产品心智冲突。根源解法是把 `Tab` / `Shift-Tab` 也收口到 `AppDelegate` 的本地键盘监视器里统一路由到 ViewModel；搜索框层只保留同样的回调语义，避免焦点路径分叉。
- **防止面板关闭状态污染**：面板彻底关闭（`hidePanel`）时强制置 `isContinuousPasteEnabled = false`。
- **文件剪贴板不是“单路径”而是“一次复制的完整文件集合”**：Finder 多选复制时不能只保存第一个文件，否则后续粘贴、打开、预览都会悄悄丢失上下文。当前做法是把整组文件路径编码为一条 `file` 记录，在监控、列表文案、预览、打开和回写剪贴板时统一按多路径语义处理。
- **文件条目的 `⌘O` 不是“逐个直接打开”而是“按整组 Reveal in Finder”**：对剪贴板里的文件集合，用户更需要回到 Finder 继续确认/拖拽/操作整组上下文，而不是把每个文件都直接 launch。当前做法是对 `file` 类型统一使用 `NSWorkspace.activateFileViewerSelecting(...)`，让单文件和多文件都按 Finder 选中语义打开，URL 仍保持普通 open 行为。
- **`Esc` 必须遵循”先回退上下文、再退出面板”的 launcher 心智**：如果用户正在搜索或按类型筛选，直接关闭面板会打断探索路径，也不符合 Raycast / Spotlight 一类控制中心的肌肉记忆。当前做法是把 `Esc` 收口为统一分层回退：即时清空搜索词和类型筛选回到完整列表，只有在已经回到默认列表时才真正关闭面板，而且该行为由 `AppDelegate` 全局兜底，不依赖搜索框焦点。
- **动作面板是静态命令表，不再内建搜索框**：Clipin 的动作数量小且结构稳定，继续维持 palette 内搜索只会引入额外模式、输入法路径和主搜索框泄漏风险。当前做法是把 `⌘K` 面板收回成静态 command sheet，只保留上下选择、回车执行和 `Esc` 关闭；普通字符、退格和 `Tab` 会在 palette 打开时被吞掉，防止输入回流到主搜索框，但不再维护独立 `actionQuery`。
- **动作面板必须是“全局命令入口”，不能因为空列表或未选中条目而失效**：如果 `⌘K` 只有在存在选中项时才可用，那么空态、筛选无结果或刚清空历史时用户会失去唯一的键盘命令出口，这和 Raycast/launcher 的稳定心智冲突。当前做法是把 action palette 拆成“依赖选中项的条目动作”和“始终可用的全局动作”两层；即使没有选中条目，也仍然提供设置、连续粘贴切换、清空筛选等命令。
- **全局命令入口不仅要可用，还要持续可发现**：如果空历史、无选中项或筛选无结果时把 `⌘K` 提示从底栏/空状态里藏起来，用户会误以为动作面板暂时不可用，已有能力也会退化成“隐形功能”。当前做法是在底栏始终保留 `⌘K` 的 Actions 徽标，并在空状态文案里直接提示可继续打开动作面板执行全局命令。
- **首次进入引导必须是独立 welcome scene，但不能变成升级骚扰，也不能因为中途关闭就永远消失**：真正的新安装需要一个单独的 onboarding window，把产品气质、主快捷键和辅助功能授权收进同一条欢迎路径里；但现有用户绝不能因为版本更新突然被欢迎页打断，新用户也不能因为首启已创建本地存储、下次启动就被误判成老用户。当前做法是把 onboarding 做成版本化状态，并在第一次看到 onboarding 功能时持久化“`newInstall` / `legacyInstall`”归类：只要首次判定命中“预存储目录 / 历史记录 / 已有偏好 / 已授权权限”等 legacy 信号，就把该安装记为 legacy 并直接视为已完成；否则记为 new install，并在 `settings.onboardingVersion` 完成前持续展示。
- **首次引导测试入口要低干扰，但需要在 release 里也能自助复现**：onboarding 的视觉和状态机需要反复验证，而且发布后的真实环境也可能需要手动重看/重置首启状态。当前做法是不再把入口限制在 `#if DEBUG`；而是把“立即重看 onboarding”和“清空 onboarding 完成状态”收进状态栏右键的 `Onboarding` 子菜单，以及 app 顶部菜单栏的 `Onboarding` 次级菜单里，保持可达，同时尽量不打扰普通主路径。
- **首次引导必须是真正的键盘路径，而不是“看起来像 launcher、却只能靠鼠标点”**：如果 welcome scene 还要用户用鼠标追按钮，就和 Clipin 的产品承诺自相矛盾，也更容易在中英文布局切换时暴露标题截断、胶囊挤压和底栏信息缺位。当前做法是把 onboarding 的左右切页、Return 默认动作和 Esc 返回都收口到统一键盘语义里，同时把 hero 区和快捷键胶囊改成自适应布局，避免中文标题和长文案被横向压扁。
- **没有辅助功能权限时，主面板应保持可见并把失败路径显式转成授权路径**：对 Clipin 来说，未授权时“先关面板再静默失败”比直接报错更伤用户心智，因为它会让用户误以为 paste 成功了。当前做法是在执行 paste 前先校验辅助功能权限；若未授权，就保留 launcher，不继续 hide panel / simulate paste，而是复用同一个权限窗口把用户带去系统授权。首次引导的权限页也不再提供“稍后再说”，必须授权后才能完成 onboarding。
