# Clipin

现代化 macOS 剪贴板管理器。Rust 后端 + SwiftUI 前端。

## 架构

- **Rust Core**（`rust/`）：存储（SQLite+FTS5）、搜索、数据模型
- **Swift UI**（`Clipin/`）：SwiftUI 界面、系统集成（menu bar、快捷键、剪贴板监控）
- **UniFFI Bridge**：Rust ↔ Swift 互操作，自动生成绑定代码

## 构建

```bash
# 首次构建 Rust + 生成 Swift bindings
./scripts/build-rust.sh

# 生成 Xcode 项目
xcodegen generate

# 编译
xcodebuild -project Clipin.xcodeproj -scheme Clipin -configuration Release build

# Rust 测试
cd rust && cargo test --lib
```

## 关键路径

- `rust/src/lib.rs` — Rust 入口 + UniFFI 导出（ClipinCore API）
- `rust/src/storage.rs` — SQLite + FTS5 存储层（v4：含 ocr_text 列）
- `Clipin/App/AppDelegate.swift` — Menu bar + NSPanel + 生命周期
- `Clipin/Views/MainPanel.swift` — Raycast 风格双栏布局
- `Clipin/Services/ClipboardMonitor.swift` — 剪贴板轮询
- `Clipin/Services/HotKeyService.swift` — ⌘+Shift+V 全局快捷键
- `Clipin/Services/PasteService.swift` — 模拟 Cmd+V 粘贴
- `Clipin/Generated/` — UniFFI 自动生成（.gitignore）
- `scripts/build-rust.sh` — 构建脚本
- `project.yml` — xcodegen 配置

## 决策

- **拼音搜索**：DB 层存 `pinyin_flat`（"你好"→"nihao"）和 `pinyin_initials`（"nh"）两列，写入时由 `compute_pinyin()` 计算（`pinyin` crate），FTS5 trigram 索引覆盖，≥3 字走 FTS 自动匹配，≤2 字走 LIKE 回退同时覆盖两列。调用 `(&*s).to_pinyin()`（对 &str，不对 String）。Migration v5 三阶段：SQL schema → Rust 回填 → FTS 重建（不可合并，中途需释放锁）。
- **搜索排序**：FTS 路径 `ORDER BY is_pinned DESC, paste_count DESC, clip_fts.rank, copy_count DESC, created_at DESC`；`clip_fts.rank` 是 BM25 负值（越小越相关）。LIKE 路径 `ORDER BY is_pinned DESC, paste_count DESC, copy_count DESC, created_at DESC`。搜索 LIMIT 统一为 200。`paste_count`（用户主动粘贴次数）是首要信号，`copy_count`（同内容被复制次数）作为次级信号；两者语义不同，不合并。
- **paste_count 追踪**：DB v6 migration 添加 `paste_count INTEGER NOT NULL DEFAULT 0`。`performPaste` / `performPastePlain` 成功写入剪贴板后调用 `incrementPasteCount(id:)`（fire-and-forget，不影响粘贴流程）。Footer badge 优先显示 "Pasted N times"，无粘贴记录时回退为 "N copies"。
- **IME 实时搜索**：`NSControlTextDidChangeNotification`（NSTextField 层）在 IME 组词（`setMarkedText`）时不触发，只在文字提交后触发。修复：在 `controlTextDidBeginEditing` 里对 field editor（NSTextView）注册 `NSText.didChangeNotification`，preedit 期间 `editor.string` 包含组词文字，直接更新 `parent.text` 触发实时搜索。`controlTextDidEndEditing` 时移除 observer，防重复注册。
- **图片 OCR**：使用 Apple Vision Framework `VNRecognizeTextRequest`（macOS 原生，零依赖，支持中英文）。图片写盘后在同一 `Task.detached` 内串行 OCR，结果写入 `clip_items.ocr_text` 列。FTS5 索引同步覆盖 ocr_text，搜索图片文字与搜索文本完全统一。列表 preview 通过 SQL 层 `COALESCE(NULLIF(ocr_text,''),content)` 智能回退，无需改 Swift 模型。
- **Bridging Header**（非 modulemap）：解决 Xcode Explicit Module Build 下 UniFFI C 头文件导入问题
- **xcodegen**：代码化管理 Xcode 项目，.xcodeproj 不提交
- **LSUIElement=true**：纯 menu bar app，不出现在 Dock
- **AppState: `@unchecked Sendable`**（非 `@MainActor`）：ClipinCore 内部用 Mutex 保证线程安全
- **AppDelegate: `@MainActor`**：所有 UI 交互在主线程，避免 Swift 6 actor isolation 错误
- **不沙盒**：CGEvent tap 需要辅助功能权限，沙盒限制太多
- **隐私感知采集**：`ConcealedType`（密码管理器）始终跳过，`TransientType` / `AutoGeneratedType`（临时/自动生成）由设置开关控制（默认跳过）。前者是不可关闭的安全红线，后者是降噪偏好。
- **所有辅助窗口（设置、引导、权限）均不用 `.floating`**：`.floating` 会让窗口悬浮在所有 app 上方——设置窗口会导致 Cmd+W 发给其他 key window，引导/权限窗口会遮挡用户需要操作的 System Settings。正确做法是普通 window level + `NSApp.activate` 让 Clipin 成为 frontmost，窗口随 app 焦点切换自然退到背后。System Settings 通过 `NSWorkspace.OpenConfiguration.activates = true` 显式置前，不依赖 floating 来解决层级问题。
- **主面板、动作面板、设置页和权限窗口都必须共享同一套 surface/control/selection 语法，不要各自长出独立皮肤**：真正要统一的不只是颜色 token，还包括 `section gutter / row chrome / content stage` 这套布局骨架。当前做法是把统一 chrome 下沉成共享 `ClipinSurfaceBackground` / `ClipinSelectableRowBackground` / `ClipinKeycap` 原件；主列表向动作面板看齐，先给 section 留 gutter，再在里面放 selected row；preview 右侧保留一层安静的内容 stage 和一个轻量 metadata block，但这两者必须是专用 surface role，而不是继续复用设置页那种 generic grouped 面，避免 detail 区重新长回“卡片套卡片”。
- **”不要贴边”有三层根因需同时解决**：①`listScrollbarInset` 造成 sidebar row 选中高亮左右不对称（左 22pt / 右 30pt），根治做法是删除该常量、改为对称 `.padding(.horizontal, listContentInset)`；②`shellSectionInset` 不足导致 section 面板嵌在 shell 角度曲线内侧（16pt < shellCornerRadius=24），调整为 18pt 让面板离 shell 边缘更有呼吸感；③嵌套圆角比例失衡（section=20 过大），按 shell 24 → section 16 → contentStage 14 → metadata 12 的层级依次递减，确保内层对象的角度曲线不与外层冲突。同时 `sectionHeader` 的 `.padding(.horizontal, 6)` 改为引用 `listRowOuterInset` token，与 row 背景左边缘对齐，消除视觉上的错位感。
- **底栏是固定高度的 command strip，不能再让 Paste CTA 反向决定整体高度**：之前 footer 的高度被左侧主胶囊按钮撑大，导致有选中项和无结果时气质不统一。当前做法是先定义 strip 自己的最小高度和外边距，再把 CTA 收进这个高度里，同时压缩 icon 和垂直 padding，让 footer 始终像一条命令条而不是一颗大按钮拖着一整条底栏走。
- **preview contentStage 比 metadata 层级更高，用投影而非尺寸差来表达**：两者 surface role 相同但语义不同（内容主角 vs 元数据注脚）。给 `contentStage` surfaceStyle 加 `shadow(radius=12, y=4)`，metadata 维持无阴影，让 preview 在视觉上”浮起”。不改 objectInset，避免用空间差代替投影差。
- **全局间距用单一 `shellGap=8pt` token 控制**：之前 shell→section / section→section / 垂直节奏有 8 个碎片 token（shellSectionInset/Gap、headerXxx、footerOuterXxx、contentXxx）全是 12pt，用户无法”改一个值调整所有间距”。现在统一为 `ClipinChrome.shellGap`，`listRowOuterInset=8pt` 同步作为选中背景到 section 边缘的固定 gutter。
- **列表型对象 row gutter 的 `.background`/`.padding` 顺序决定背景宽度**：`.padding(.horizontal, X).background(...)` 让 background 包含 padding（background 宽 = content + padding）；`.background(...).padding(.horizontal, X)` 让 background 只包含 content，padding 在背景外侧。三处列表（主列表、设置 sidebar、动作面板）必须统一为后者，让 background 贴内容、外部 `listRowOuterInset=8pt` 提供 gutter，否则视觉宽度不一致。
- **PreviewPane 的 contentStage/metadata 卡片不加任何外层垂直 padding**：上方呼吸感由卡片内部的 `detailStageInset` 体现，下方对齐由 footer 的 `shellGap` 外间距负责。曾试过 `.padding(.vertical, shellGap)` 结果 metadata 底部和 column 底边多出 8pt 缝隙，与 sidebar 背景不对齐——`.padding(.vertical)` 上下语义不同，正确做法是不加。
- **设置窗口所有入口都必须落到同一个自定义 NSWindow，而不是一部分走 SwiftUI Settings scene、一部分走 AppDelegate 窗口**：否则会出现首击行为、鼠标交互、键盘上下文和窗口样式不一致，看起来像“设置页又坏了”。当前做法是把 `Command+,` 的 `.appSettings` 命令显式重定向到 `AppDelegate.openSettingsWindow()`，保留自定义 `ClipinHostingView` 的首次点击行为，彻底消除系统 Settings scene 和自定义窗口并存带来的交互分叉。
- **连续粘贴模式**：面板保持可见，失去 key window 时自动夺回焦点（通过 `ClipinPanel.resignKey` 回调 + 150ms 延迟让用户点击先完成）。用户可以在不同应用的输入框间点击定位光标，面板自动回焦后直接上下选择、回车粘贴，实现零额外操作的连续粘贴。通过 `didActivateApplicationNotification` 实时更新底栏目标应用名，粘贴时 `resolveTargetApp()` 实时查询 `frontmostApplication` 兜底。Esc 退出连续粘贴模式。常驻可见（不夺回焦点、可编辑浮动笔记）作为后续 feature。
- **全局统一导航快捷键必须按窗口上下文路由，而不是默认主面板永远优先**：之前把 `↑↓ / Return / Esc / Tab` 一股脑收口到主面板，导致设置页拿不到方向键，键盘体验在不同窗口之间断裂。当前做法是在 `AppDelegate` 的本地键盘监视器里先判断当前 key window 所属上下文（主面板 / 动作面板 / 设置页），再把导航分发给对应模型：主面板继续负责列表导航和粘贴，设置页通过独立的 `SettingsNavigationModel` 响应 `↑↓` 切换 sidebar tab，避免再靠 `List(selection:)` 或局部控件焦点偷带键盘语义。**设置页 sidebar 用 ScrollView+LazyVStack 而非 List**：macOS `List(.sidebar)` 底层是 NSTableView，系统原生处理 ↑↓，但这是隐式行为，换实现就会悄悄消失（踩过两次）。必须始终从 key monitor 显式处理 ↑↓ → `SettingsNavigationModel.selectNext/selectPrev`，让键盘逻辑与渲染实现彻底解耦。
- **类型筛选切换也必须是全局导航，不应依赖搜索框焦点**：只在 `SearchBar` 内部拦截 `Tab` 会导致用户点击列表/预览区后，`Tab` 退化成默认焦点遍历，和“键盘优先”的产品心智冲突。根源解法是把 `Tab` / `Shift-Tab` 也收口到 `AppDelegate` 的本地键盘监视器里统一路由到 ViewModel；搜索框层只保留同样的回调语义，避免焦点路径分叉。
- **`Space` 预览也必须是全局导航，但 IME 组词时必须放行**：有可预览选中项时触发 Quick Look，无可预览项时吞掉按键。例外：当 `NSTextView.hasMarkedText() == true`（拼音等输入法正在选字），必须将事件交还给 IME，否则空格确认字符失效。key monitor 在 IME 处理之前拦截事件，`hasMarkedText` 检测必须在这一层做。
- **防止面板关闭状态污染**：面板彻底关闭（`hidePanel`）时强制置 `isContinuousPasteEnabled = false`。
- **空格预览必须统一走“resolver + session + app focus state”三层，而不是在 ViewModel 里临时拼 URL 后直接弹窗**：真正要统一的不只是 `Space` 这个快捷键，还包括底栏可见性、动作面板里的 Preview 命令、Quick Look 的文件存在性校验，以及连续粘贴模式下的焦点恢复。当前做法是把“这个条目能否预览、预览哪些 URL”收口到 `ClipPreviewResolver`，把 Quick Look 生命周期收口到 `QuickLookPreviewService`，再由 `AppDelegate` 监听 preview 可见性统一处理连续粘贴的夺焦暂停与恢复，避免同一能力在 ViewModel / Service / AppDelegate 三处各算一遍、彼此打架。
- **Quick Look 预览必须以”当前可见列表”为 session，而不是只预览当前条目**：系统预览面板的默认心智就是在一组 item 里连续浏览；如果只把当前条目的 URL 喂给 Quick Look，用户会直接失去预览中的上下切换，也会让主列表选中和系统面板脱节。当前做法是由 `ClipPreviewResolver` 基于 `flatOrder` 生成 preview session，并由 `QuickLookPreviewService` 用索引变化反向同步 launcher 选中项；这样 `Space` 进入预览后，仍然能沿当前筛选结果继续浏览。QLPreviewPanelDelegate 的 `handle event:` 方法中，方向键自带 `.numericPad`/`.function` flag，必须只过滤 shift/ctrl/opt/cmd（`significantFlags`），不能用 `flags.isEmpty`。
- **列表分页加载（无限滚动）**：初始加载 50 条（`pageSize=50`），`LazyVStack` 末尾放透明 sentinel，`onAppear` 时调用 `loadMoreItems()` 追加下一页。`hasMore` 为 `false` 时不渲染 sentinel，避免重复触发。分页只对普通视图（非 pinned、非搜索）有效；搜索返回全部匹配结果，pinned 视图条目数量少，均不需要分页。
- **文件剪贴板不是“单路径”而是“一次复制的完整文件集合”**：Finder 多选复制时不能只保存第一个文件，否则后续粘贴、打开、预览都会悄悄丢失上下文。当前做法是把整组文件路径编码为一条 `file` 记录，在监控、列表文案、预览、打开和回写剪贴板时统一按多路径语义处理。
- **文件条目的 `⌘O` 不是“逐个直接打开”而是“按整组 Reveal in Finder”**：对剪贴板里的文件集合，用户更需要回到 Finder 继续确认/拖拽/操作整组上下文，而不是把每个文件都直接 launch。当前做法是对 `file` 类型统一使用 `NSWorkspace.activateFileViewerSelecting(...)`，让单文件和多文件都按 Finder 选中语义打开，URL 仍保持普通 open 行为。
- **`Esc` 必须遵循”先回退上下文、再退出面板”的 launcher 心智**：如果用户正在搜索或按类型筛选，直接关闭面板会打断探索路径，也不符合 Raycast / Spotlight 一类控制中心的肌肉记忆。当前做法是把 `Esc` 收口为统一分层回退：即时清空搜索词和类型筛选回到完整列表，只有在已经回到默认列表时才真正关闭面板，而且该行为由 `AppDelegate` 全局兜底，不依赖搜索框焦点。
- **动作面板是静态命令表，不再内建搜索框**：Clipin 的动作数量小且结构稳定，继续维持 palette 内搜索只会引入额外模式、输入法路径和主搜索框泄漏风险。当前做法是把 `⌘K` 面板收回成静态 command sheet，只保留上下选择、回车执行和 `Esc` 关闭；普通字符、退格和 `Tab` 会在 palette 打开时被吞掉，防止输入回流到主搜索框，但不再维护独立 `actionQuery`。
- **动作面板必须是“全局命令入口”，不能因为空列表或未选中条目而失效**：如果 `⌘K` 只有在存在选中项时才可用，那么空态、筛选无结果或刚清空历史时用户会失去唯一的键盘命令出口，这和 Raycast/launcher 的稳定心智冲突。当前做法是把 action palette 拆成“依赖选中项的条目动作”和“始终可用的全局动作”两层；即使没有选中条目，也仍然提供设置、连续粘贴切换、清空筛选等命令。
- **全局命令入口不仅要可用，还要持续可发现**：如果空历史、无选中项或筛选无结果时把 `⌘K` 提示从底栏/空状态里藏起来，用户会误以为动作面板暂时不可用，已有能力也会退化成"隐形功能"。当前做法是在底栏始终保留 `⌘K` 的 Actions 徽标，并在空状态文案里直接提示可继续打开动作面板执行全局命令。
- **圆角透明的根治方案**：SwiftUI `.clipShape()` 只裁 SwiftUI 渲染树，`NSVisualEffectView`（`.regularMaterial` 底层）走 AppKit compositor 渲染，不受 SwiftUI mask 约束。必须在 `ClipinHostingView.updateLayer()` 里设 `layer.masksToBounds = true` + `layer.cornerRadius = shellCornerRadius`，让系统 compositor 在 CALayer 层裁掉所有 subview。`hasShadow = true`：corners 真正透明后，AppKit shadow 跟随 alpha 轮廓，不再是矩形。SwiftUI `.shadow()` 在 `masksToBounds = true` 下会被裁掉，不要用。浅色模式下 border 改用 `Color.primary.opacity(0.12)`（深色内描边），白色描边在白色背景上不可见。
- **设置窗口复用时不 `center()`**：只在首次创建时居中（`isNew` 分支），复用已有窗口时保留上次用户移动的位置，符合 macOS 所有系统偏好窗口的惯例。
- **权限提示只在粘贴时按需触发，不在启动时主动弹出**：启动时弹权限窗会在用户还没用到粘贴功能时就打断体验；权限缺失的后果（粘贴失败）本身就是最清晰的上下文，按需触发更自然。权限提示入口只有一处：`executePasteFlow` 检测到无权限时。
- **onboarding 权限步骤必须有"稍后再说"退出路径**：让用户不选择授权就能继续用 Clipin 的历史记录功能，不能用没有出口的半模态锁住用户。"稍后再说"调用 `flow.skipPermission()` → `finishOnboarding()`，行为等同于完成引导、不请求权限。
- **launcher 的浏览模式必须收成单一 `browseMode` 状态，而不是继续维护 `typeFilter + isPinnedView` 两套互斥分支**：类型 pill、固定视图、默认入口、Esc 回退、Tab 循环和“记住上次视图”本质上都在描述同一件事：当前用户在浏览哪个列表。当前做法是引入 `LauncherBrowseMode`（all / pinned / text / image / file / url），由 SearchBar、ViewModel 和 AppDelegate 统一读写，避免 pinned 视图和类型筛选在不同层各自长出一套状态机。
- **`pinned` 的普通浏览呈现和 launcher 默认视图都必须进入设置，而搜索始终是全局语义**：普通浏览里，固定项是混排、顶部分组还是只在 pinned 视图中出现，属于用户偏好，不应被写死；打开 launcher 时进全部、进 pinned 或记住上次，也是入口偏好。当前做法是在 `SettingsStore` 中增加 `PinnedItemsPresentation` 和 `LauncherDefaultView` 两个枚举，并持久化 `lastLauncherBrowseMode`。搜索时忽略 pinned 浏览策略，只保留内容类型筛选，确保任意条目都能被稳定召回。
- **用户可见文案不能只本地化 SwiftUI 的静态标题，菜单、动态 notice、状态文案和快捷键名也要统一走 `Localizable.strings`**：如果 `NSMenuItem.title`、`String(format:)` 提示、文件摘要或 `HotKeyService.displayKey` 继续直接返回英文，中文环境会立刻露馅。当前做法是把所有静态 key 收口到 `Localizable.strings`，并让动态提示统一通过 `NSLocalizedString + String(format:)` 组装，菜单和服务层同样遵守这条路径。
- **设置页这类 SwiftUI 表单封装必须保住 `LocalizedStringKey`，不能在 helper 签名里退化成 `String`**：一旦 `settingFieldRow / toggleSettingRow / actionRow` 先把字面量参数收成普通字符串，后面的 `Text(title)` 和 `Button(buttonTitle)` 就不会再触发本地化查找，结果是同一个页面里部分区域翻译、部分区域退回英文。当前做法是 helper 层统一接收 `LocalizedStringKey`，只有运行时动态 note/notice 才保留 `String`。
- **当普通浏览隐藏 pinned 时，分页必须补齐“可见的一页”，而不是把原始 SQL 页直接交给前端过滤**：否则一旦历史顶部被 pinned 塞满，第一页会被全部隐藏，用户看到的就是错误的空列表。当前做法是在 ViewModel 中按 raw offset 连续拉取多页，直到凑满一页可见项或到底，再决定 `hasMore`，保证无限滚动和用户看到的列表是同一套语义。
